## Feature Boundary

This feature turns the already-landed two-stage runtime contract into the default packaged production model path by shipping, validating, and activating one canonical composite bundle for new installs while preserving explicit binary compatibility overrides for migration and recovery.

**Domain type(s):** RUN | CALL | ORGANIZE

---

## Locked Decisions

These are fixed. Planning must implement them exactly. No creative reinterpretation.

### Production Default
- **D1** For new `full-stack same-host` packaged installs, the default production model path must be a composite two-stage bundle, not the legacy binary bundle.
  *Rationale: The runtime contract and operator-surface semantics already exist; productization means making the intended path the actual shipped default.*

- **D2** `console-only` mode does not require model activation and therefore does not block on the two-stage bundle being active.
  *Rationale: The control-plane-only install path should remain independent of sensor/runtime activation concerns.*

- **D3** The legacy binary bundle remains supported only as an explicit compatibility or recovery override, not as the silent default for new packaged installs.
  *Rationale: Migration support is necessary, but defaulting new installs to the old path defeats the productization goal.*

### Bundle Contract
- **D4** Production must continue to use one activation record and one composite bundle that carries both stage 1 and stage 2 together.
  *Rationale: The single-activation contract is already locked by the runtime lane and prevents stage drift.*

- **D5** The shipped default two-stage artifact must be generated by the canonical packaging flow, not by manual manifest editing or post-hoc file surgery.
  *Rationale: The current digest mismatch failure is exactly the kind of artifact drift this feature must eliminate.*

- **D6** The default shipped composite bundle must live under the packaged `artifacts/final_model/` surface and be the bundle the `full-stack same-host` installer uses automatically when valid.
  *Rationale: The operator story must be concrete and discoverable, not dependent on out-of-band artifact hunting.*

- **D7** Stage-2 abstention thresholds, label set, and family semantics remain embedded in the versioned composite bundle contract, not in Linux host config.
  *Rationale: This is model behavior and must ship with the model, not with the host.*

### Runtime Semantics
- **D8** When the composite bundle is active, runtime keeps all existing binary outputs unchanged and emits family enrichment additively by default.
  *Rationale: Productization should activate the already-designed semantics, not redesign the output contract.*

- **D9** If the stage-2 side of the composite bundle is missing, incompatible, or fails validation, activation/startup must fail closed; there is no silent fallback to binary-only behavior under the same default activation path.
  *Rationale: Silent downgrade would hide contract drift and make operator expectations unreliable.*

### Packaging & Release
- **D10** Release creation must fail if the default shipped composite bundle cannot pass the same validation prerequisites required for activation.
  *Rationale: A packaged default that cannot be verified is not a product artifact; it is a broken build.*

- **D11** The default shipped two-stage bundle should be treated as the canonical operator-facing artifact for new packaged installs, while explicit override flags remain the migration escape hatch.
  *Rationale: This preserves operational control without forcing every install to behave like a migration edge case.*

### Scope Control
- **D12** This feature only productizes the bundled default model path. It does not reopen the already-closed questions about direct multiclass replacing the binary gate.
  *Rationale: The evidence-backed direction is already locked: binary gate first, family enrichment second, abstain to `unknown` when weak.*

### Agent's Discretion
- Exact bundle name/version suffix and composite artifact naming.
- Exact build/install sequencing between bundle packaging, release validation, and activation.
- Exact fallback/override UX for operators who intentionally want a legacy binary bundle.
- Exact test matrix that proves packaged two-stage readiness.
